# 分阶段地编流程

搭建或改造关卡时读取。每个阶段只解决一类空间问题，并在 Gate 通过后继续。

## Stage 0：安全与基线

- 确认目标 World、关卡路径、允许修改范围、PIE 状态和保存策略。
- 读取 World Partition、Data Layers、Landscape、Water、道路、PCG/Foliage、关键 Actor 和灯光。
- 验证承载表面：检查 Landscape 组件与高度采样，并向关键区域做向下射线；任一失败都不得假设可以贴地或进入 PIE。
- 捕获俯视、玩家视角和必要的带网格/标签视口图。
- 若处理正式地图，使用用户指定副本或 Data Layer；探索稿使用隔离目录。

**Gate：**目标路径明确；编辑器不在 PIE/Simulate；已有资产不会被无意覆盖；地表有效，或已明确采用测试底盘；已有基线数据与截图。

## Stage 1：尺度、边界与灰盒

- 依据玩家胶囊、镜头、移动速度和现有模块确定空间尺度。
- 建立可玩边界、主要高度层、体块、关键地标占位和不可达区域。
- 使用简单可替换几何；给 AI 生成对象加阶段标签或放入专用 Data Layer。

**Gate：**预期通道可通过；主要区域和高差层级在俯视图中可读；灰盒对象可追踪、可隐藏且不污染正式命名。

## Stage 2：动线、回环与 POI

- 先放主路线，再放备选路线和回环；明确瓶颈、制高点、风险区和安全区。
- 按体验节奏布置 POI、出生点、目标点和远景地标。
- 用路径长度与当前移动速度估算行进时间，避免只看平面距离。
- 用空间、灯光、颜色和几何可供性传达可走、可跳、可交互与危险区域；不要依赖文字或小地图完成基础教学。
- 战斗入口保留读场窗口，使玩家能观察主要威胁、掩体、侧翼和撤退方向。
- 对战地图检查路线公平性、视线和交叉火力；探索地图检查引导、揭示节奏和回返成本。

**Gate：**关键目标有符合意图的到达方式；死路是有意设计；地标支持方向判断；关键视线、瓶颈和高地有证据；玩家进入遭遇前有机会读场。

**白盒试玩 Gate：**进入生态、正式资产和视觉包装前，用真实试玩验证关键路径、节奏与路线选择；记录关键路径完成情况、计划与实测时长偏差、实际采用路线。未通过时先调整站位、路线、高差、视线和资源位置，再考虑敌人数值，并继续迭代白盒。阈值按项目设定，不套用固定数字。

## Stage 3：地形、水体与道路

- 让地形服务玩法：遮挡、路线、制高点、盆地、边界和远景，而非只追求随机起伏。
- 先处理大形，再处理中形；微表面细节留到结构稳定后。
- 道路连接 POI 并顺应坡度；水体明确岸线、跨越点、阻挡和导航影响。
- 在正式建筑前建立“道路 → 地块 → 建筑 → 道具”关系，定义地块出入口、前后场、退界、停车/装卸/院落和边界。
- 道路端点必须接入网络、形成有意死路/回车区、通过遮挡延伸出边界，或由场景边界明确收口；不得裸停在测试底盘。
- 使用样条时检查连接、切线、坡度、交叉口、桥梁和地形穿插。

**Gate：**不破坏 Stage 2 的动线；道路端点和主要地块关系可解释；无明显漂浮、埋入、断裂样条或不合理坡度；Landscape Bounds、图层和保存状态正确。

## Stage 4：生态、植被与 PCG

- 先定义生物群系和禁放区，再生成植被；道路、水面、建筑、战斗空间和关键视线优先清空。
- 用密度、尺度、随机种、坡度、高度和遮罩控制分布。
- 把近景可读性、中景层次、远景轮廓分开处理。
- 大规模实例使用 Foliage/PCG/HISM 等合适机制，避免用大量独立 Actor 代替实例。

**Gate：**禁放区无实例；关键路线和地标未被遮死；分布有生态逻辑且可复现；实例规模符合目标平台。

## Stage 5：建筑、道具与视觉叙事

- 先做地标和轮廓，再做功能性物件、掩体、引导物和故事细节。
- 批量摆放前试摆代表资产，确认真实 Class、正面轴、Pivot 和尺寸；不要根据资产名前缀猜类型。
- 使用陌生或大批资产集时，先在隔离 Asset Zoo（资产陈列测试关卡）中按类别网格排开并加入尺度锚点，走查 Bounds、轮廓、材质/光照和碰撞，再进入正式场景。
- 批量摆放后以人物/门、椅子/桌子、车辆等可信对象为尺度锚点，按建筑、家具和小型道具比较世界 Bounds，标记违反常识的尺寸离群项；`Scale = 1` 不能证明尺寸正确。
- 使用模块化套件与实例，遵循项目 Pivot、网格和命名规范。
- 对模块套件做回环、堆叠和缝隙压力测试，覆盖内外角、门窗、多层、斜接与碰撞；不要只验证理想拼法。
- 模块建筑优先使用网格/顶点吸附，贴坡道具使用表面吸附并控制法线旋转与偏移；吸附成功后仍需验证 Bounds、接触面和局部视图。
- 放置网格后用世界 Bounds 对齐目标中心和底面；道路等平面允许 Z Extent 为 0，但 X/Y Bounds 必须有效。
- 对新增对象执行 3D Bounds 粗筛：整栋建筑间的非设计性重叠作为硬冲突；模块接缝允许候选，但必须有收口且无可见穿模、Z-fighting 或碰撞阻断；建筑—道具/绿化候选用局部透视确认。
- 每栋主要建筑都要能解释功能、临街面、入口路径、后场和相邻边界；车辆相关建筑还要解释车道、停车或装卸位置。
- 把相关道具组成有功能或叙事关系的簇并保留负空间，不平均撒点；战斗区域清理人物高度附近遮挡角色轮廓、交火角或常用瞄准区的非必要杂物。
- 建立密集区、过渡区和疏松区；进入可信场景或展示级前，清理大面积裸露测试地面、无意孤立资产和机械等距排布。
- 检查碰撞、贴地、穿插、重复图案和可交互可达性；比较相邻资产的纹理尺度，并复核 UV 拉伸、接缝与明显平铺周期。

**Gate：**焦点清晰；装饰不破坏移动、战斗或交互；整栋建筑间无非设计性重叠，模块接缝已收口，其他候选已复核；无明显漂浮、穿模、错误比例、材质尺度错乱和高频重复；对象可读回。

## Stage 6：灯光、雾与可读性

- 先建立时间、天气、主光方向和曝光，再用局部光强化路线、入口、目标与层次。
- 同时检查玩家视角、俯视/远景和不同朝向；避免只把一个截图角度调好。
- 保持 gameplay read：交互物、危险区、可走区域与背景有足够分离。

**Gate：**关键目标和路径可读；曝光、雾和阴影没有吞掉导航信息；截图已经实际查看。

## Stage 7：运行、流送与性能

- 进入 PIE 前保存；检查出生、碰撞、穿模、导航、触发、载具和相机。
- 先确认 PlayerStart 与关键路线下方射线能命中可碰撞表面；失败时跳过 PIE，并明确记录缺失前置条件。
- World Partition 地图检查相关 Cells/Data Layers 的加载与卸载。
- 大世界或高密度场景先获取 Game/Render/GPU 帧耗和瓶颈判定，再决定优化方向。
- 检查日志中的 error、warning、partial、compiler errors 和资源缺失。

**Gate：**PIE 能完成关键路径；无阻断性日志错误；流送无明显空洞；性能结论有数据。

## Stage 8：交付

- 保存目标关卡和本轮承诺资产。
- 输出创建/修改/跳过/失败清单。
- 给出基线与完成状态的可见差异、Gate 结果和遗留风险。
- 标注哪些是灰盒、哪些可继续迭代、哪些尚未进行美术或性能验证。
