# MCP 镜头执行桥接

本文件把“设计师分镜执行规格”桥接到“UE MCP / VibeUE 执行”。它不是底层 K 帧手册；底层 LevelSequence 创建、Actor 绑定、Transform 通道、保存和验证交给下游 VibeUE Sequencer 执行规范。

## 何时进入 MCP 执行

用户出现以下意图时，进入执行流程：

- “直接做”
- “在 UE 里创建”
- “新建定序器”
- “做一个镜头/过场/动画”
- “帮我 K 一段”
- 指定了场景 Actor、地图区域、资源路径或 LevelSequence 路径
- 提供了分镜、镜头表、分段画面描述或脚本，并要求快速动态

如果用户只是在讨论创意、分析镜头、写分镜，不执行 UE 修改。

## 执行模式

根据用户意图选择最低可满足的模式。用户明确指定时以用户指定为准；未给模式名称时，按“构图确认→静态分镜、单镜头/单技术点→快速单镜头、测试/试做/先跑通/粗版/先看看→快速动态、正式/标准/制作级/完整制作→标准制作、已有连续影像并要求验收→成片检查”推断。只有模式差异会显著改变写入范围、耗时或验证成本且无法判断时才询问。

### 执行模式验收矩阵

| 模式 | 适用范围 | 最低写入 | 最低验证 | 允许的完成措辞 | 升级条件 |
|---|---|---|---|---|---|
| 静态分镜 | 单镜头或多镜头构图/顺序确认 | LevelSequence、CineCamera、CameraCut、静态或极少构图 Key | 结构读回、各 Cut 代表帧构图 | `静态分镜已生成；运动未制作` | 需要可播放运动时升级 |
| 快速单镜头 | 一个 CameraCut、一个 Actor、一个运镜原语或一个技术点 | 只写实际变化的 Transform/Rig 和必要轨道；复用未变化的 CameraCut、焦距与焦点 | 一次结构读回；默认不自动截图、不导出、不全片播放 | `单镜头结构已写入；画面与运动待设计师验片` | 需要自动画面快筛、多镜头或完整修正时升级 |
| 快速动态 | 一个或多个设计师已给镜头的动态粗版，通常为多镜头 | 基础路径、朝向、焦距/焦点、CameraCut 和粗节奏 | 结构读回、低成本关键画面快筛 | `动态粗版已生成；结构/关键画面已快筛，运动待人工验片` | 需要完整曲线、避障、焦点和逐镜修正时升级 |
| 标准制作 | 设计师要求的完整镜头范围 | 完整路径、控制权、构图、曲线、参与的焦距/焦点、必要的结束状态、修正 Key 和落稳要求 | 结构、CameraCut、控制权、参与维度生效、必要状态、关键画面和风险帧快筛 | `标准制作写入完成；结构与关键画面快筛通过，运动流畅性待人工验片` | 需要交付结论或正式渲染验收时升级 |
| 成片检查 | 已有真实播放、录屏或渲染证据的交付检查 | 原则上不新增创意；只修设计师确认的问题 | 结构、画面、运动、无 UI/警告和正式输出证据 | 四层都通过才可写 `成片检查通过`；缺证据写 `成片预检完成，尚不能通过` | 缺真实证据时停留在预检状态 |

- “直接做 / 不用问”且目标足够明确时，按上面的模式映射直接执行并报告推断；不因用户没有说出模式名称停住。
- 单镜头不自动等于快速单镜头：如果用户明确要求标准制作或要求完整画面快筛，仍按标准制作执行。
- 快速动态不自动等于多 CameraCut：单镜头动态粗版需要画面快筛时也可使用；与快速单镜头的核心区别是验证深度。
- 模式只定义最低写入、最低验证和允许措辞，不规定思考步骤、工具顺序、镜头实现或检查数量；AI 可以按风险采用更高效或更充分的方法，但不得降低证据门槛或改变设计师创意。

### 变化驱动的简单运镜

缓慢推进、快速拉远、单轴横移、升降、简单摇摄/甩镜，以及开阔空间中目标和路径明确的基础环绕、跟拍、俯冲或仰冲，可按快速单镜头执行。默认一次写入、一次结构读回，并遵守：

- 已有有效 CameraCut 覆盖目标范围时直接复用；没有且镜头需要 CineCamera 输出时，只补一段覆盖范围的 CameraCut，不把它扩展成剪辑设计。
- 焦距、焦点、光圈和景深不变化且没有清晰度风险时，继承当前相机状态，不新建对应轨道，也不做未参与维度的专项验收。
- 路径远离几何体、用户已给安全区域或技术目标与遮挡无关时，不做完整场景勘察和逐帧避障；只有已知近障碍、贴地、穿门、复杂跟拍或画面证据显示风险时升级。
- 简单运动仍需要合适的基础插值，但不默认进入 Curve Editor、切线精调或连续渲染；用户反馈节奏问题或运动职责确实需要时再升级。
- 技术测试只验证当前目标。画面语义简单但不妨碍 Transform/Rig、范围或 Binding 结论时，记录局限并停止，不自动转成正式镜头返工。
- 用户明确测试 K 帧、CameraCut、Binding、Rig 或 Shake 等单项能力时，只以该技术目标和必要工程安全作为通过条件；叙事、审美、构图或成片验证只有妨碍当前结论或用户明确要求时才升级。

### 标准制作模式

标准制作把设计师分镜整理成完整 Sequencer 执行规格，检查 CameraCut、范围、重复 Key、保存、Binding、控制权、参与的焦点和必要的结束状态，并按切点、运动变化、遮挡、焦点和构图风险选择代表帧；发现明显遮挡、穿模、过曝、UI 外露、焦点无效或语义为空要修正。
#### 标准制作完成门槛

标准制作不能只证明 Sequencer 骨架存在。至少满足以下条件，才允许报告“标准制作完成”：
- 结构：LevelSequence、Playback Range、CameraCut、CineCamera 绑定、关键轨道范围、保存和读回通过。
- 镜头：每段都有与设计师意图一致的 Camera Path、参与维度的主控、构图和节奏；焦距、焦点、光圈或景深未参与变化时允许继承现状，不为完成清单强制建轨。
- 曲线：Key 密度与路径复杂度和镜头职责匹配；中间 Key 必须有避障、重锁构图、前景掠过或节奏职责，不能为了“看起来更专业”随手加。
- 可编辑性：主运动与高频有机层职责分离；Shake、Additive 或回退烘焙可单独关闭、替换和删除，主 Transform 不被周期性密集 Key 污染。
- 焦点：只有浅景深、近景、主体距离变化、焦点切换或设计师明确要求时，才要求 Focus Method、ManualFocusDistance/Tracking Focus、语义焦点目标和光圈策略写入并读回；继承固定焦点时只报告未改动，不能伪称已专项验证。
- 状态：只有角色、物体或玩法在序列结束后需要恢复或保持时，才设置并读回 Completion Mode；纯镜头属性没有结束状态要求时不做逐 Section 验证。
- 控制权：位置、朝向和焦点各只有一个主控；不存在未说明的 Transform/Rig、Look At/Rotation 或双焦点系统竞争。
- 画面：完成低成本画面快筛，确认没有黑屏、纯天空/地面/墙面、严重遮挡、穿模、过曝、UI 外露或语义为空。
- 边界：没有连续播放、录屏、渲染或等价时序分析证据时，必须使用矩阵中的标准制作措辞，不能写“运镜通过”“成片合格”或不带限定的“标准制作完成”。当前工具无法连续验片时，由设计师提供运动与主观节奏反馈。

### 成片检查模式

成片检查用于准备交付或渲染。没有真实播放、录屏或渲染证据时只能做成片预检；不能因为结构和截图通过就把状态升级为成片通过。

## 场景勘察

标准制作或成片检查前，如果核心目标、拍摄区域或路径参考不明确，先通过 MCP 做只读场景勘察，再进入 K 帧。

勘察目标：识别当前场景里可用于运镜的主要 Actor，而不是理解整个世界。优先收集：名称、类型、位置、包围盒/大小、可见性、是否像核心主体、前景层次、道路/路径参考、潜在遮挡物或背景。

执行规则：
- 不要无脑全量遍历所有 Actor；Actor 数量多时，先按名称、类型、Tag、区域、父级、前缀、当前截图方向或用户指定位置过滤。
- 优先保留大体积、可见、靠近拍摄区域、具有空间意义或轮廓清晰的 Actor；排除灯光、小型装饰、碰撞体、隐藏物、编辑器辅助物、临时调试物和明显无关对象。
- 输出足以做决定的小型候选集，复杂场景通常为 5-10 个；按“核心主体 / 前景层次 / 路径参考 / 潜在遮挡物 / 背景参照”分类，并写出大概位置和大小判断。
- 如果核心目标仍不明确，先让设计师确认；不要自行把随机大物体当成主角。
- 勘察是只读步骤，不移动、不删除、不生成场景 Actor。
## 执行前必要规格

镜头字段以 `storyboard-to-sequencer-spec.md` 为唯一宿主。进入 MCP 写入前只确认本次执行所需的最小集合：目标资产路径与同名风险、执行模式、帧率和 `[StartFrame, EndFrame)`、CameraCut、参与的 Actor/相机/轨道、位置/朝向/焦点主控，以及不可逆场景修改或结束状态风险。未参与维度不为清单完整度重复整理。

如果用户已经表达 UE 落地、Sequencer、MCP 执行、直接创建或继续制作等意图，不要停在方案。按本文件的模式映射推断并执行；先做最小目标定位和能力检查，只有主体、路径或风险不明确时才升级场景勘察，只有角色方向、交付画幅、覆盖风险或不可逆资产操作仍不明确时才提问。

空场景或专用验证场景必须额外给“最小可执行资产清单”，把必须资产和可选增强资产分开。

## 交给执行器

以下内容可由 VibeUE / UE MCP 公开封装或 UE Python / Sequencer 原生 API 处理：

- 创建或加载 LevelSequence。
- 检查同名资产，避免覆盖。
- 创建或绑定 CineCameraActor。
- 创建 CameraCutTrack。
- 绑定 Actor。
- 按参与维度写 Transform/Rig、焦距、Focus Method、手动焦点距离、景深、可见性和必要的结束状态。
- 处理关键帧通道、FrameNumber、Rotator 顺序、Scale 默认值。
- 保存资产。
- 验证 Binding、组件子 Binding、参与轨道、CameraCut、控制权和保存重载；只有存在结束状态要求时才验证 Completion Mode。

执行器根据 live schema、当前会话已验证能力、批量支持、改动风险和调用成本自主选择公开封装或原生 API，不强制先试封装。路径失败时保留已写入状态并只替换失败环节，不因技术问题改变设计师镜头。

### 传输与批量快路

- 当前客户端已暴露 Unreal MCP 工具时，必须通过原生工具调用；不得再用 `urllib`、`curl` 或临时脚本直连 `/mcp` 并手工解析 SSE。原始 HTTP 只用于连接诊断；客户端未暴露 MCP 工具时，按 `$unreal-mcp-navigator-cn` 修复连接，不通过自制传输层执行资产写入。连接无法恢复时停止 UE 写入，交付可执行规格与明确阻塞项。
- 允许根据工具能力受控委派给另一个具备 Unreal MCP 的智能体，但同一时刻只允许一个 UE 写入者；协调方在委派期间停止 UE 操作，接收方返回资产路径、实际修改和结构读回证据。口头完成声明或后台进程日志不能替代当前资产读回。
- 每个 Toolset/工具或通道类型在当前会话首次使用时读取一次 live schema 并缓存准确参数；只有 MCP 重连、schema 报错或类型变化时才重新发现，不靠反复尝试 `Location.X`、`Rotation.Y` 等名称猜接口。
- 优先使用批量工具，或在一次 `execute_python_code` 中完成同一原子步骤并集中读回；不要为每个 Key 单独发起一次 MCP 调用。只有当前工具确实只支持逐 Key 且范围很小时才逐项调用。
- 写调用超时、返回空或传输结果不明时视为“状态未知”，先读回 Sequence、Binding、Track、Section 和 Key。目标已改变则继续缺失部分；未改变才切换该失败环节，禁止直接重放整段。

## 执行时的边界

- 不默认打开 Sequencer UI。
- 不默认创建 LevelSequenceActor。
- 不默认移动、生成、删除场景 Actor。
- 如果场景中已经存在占位场景，或用户明确表示已有场景，不要重新生成/摆放场景 Actor。
- 已有场景下只允许按需求查找已有 Actor、创建/绑定 CineCameraActor、创建/修改 LevelSequence、添加 CameraCut、给摄像机和必要 Actor K 帧。
- 只有用户明确要求“重新搭场景”“补场景”“创建占位资产”时，才生成新的场景 Actor。
- 不覆盖正式资产。
- 需要遍历场景时必须给过滤条件：名称、标签、类型、区域、父级或前缀。
- 场景勘察只允许只读查询；不得借勘察之名移动、删除、重搭或生成场景 Actor。
- 如果用户要求最快处理且没有给路径，才允许临时路径，并在结果里说明。

## 成片风险预检

标准制作、成片检查，或画面风险会影响当前技术结论时，把镜头表转成可被预览检查的画面规则。单项 K 帧、Binding、Rig、Shake 等技术测试不自动扩展构图、曝光、遮挡或 UI 验收：

- 默认通过 CineCameraActor / CameraCut 预览，不用编辑器自由视角当成最终镜头。
- 规格里写明是否需要隐藏编辑器 UI、坐标轴、选择框、右下角提示和调试标记；正式预览必须隐藏。
- 强光、爆炸、技能特效、发光主体等镜头必须检查曝光风险：不要让摄像机长时间直冲纯白区域。
- 特写镜头要保留可读信息：主体可以大，但不能整段堵满画面；至少留出轮廓、环境参照或关键道具关系。
- 如果设计目标就是闪白转场，白屏必须短，并在镜头表中标注“有意闪白”，否则视为过曝问题。

### 运动镜头避障预检

创建运动镜头前，先对 Camera Path 做基础避障和黑屏风险预检：

- 按路径最近障碍、方向变化、速度峰值、构图重锁和终点职责选择风险帧，确认相机没有进入模型、贴墙、贴地、只拍天空/地面/墙面或被建筑、树、车、特效长期遮挡；没有明显风险信息时再用起点、中段、终点作低成本默认抽查。
- 能用 Line Trace 或简单碰撞检测时，检查相机到主体的视线是否被长期阻断；不能自动检测时，至少按关键帧做风险抽查。
- 前景遮挡应服务层次、速度或设计师给定意图；遮挡风险阈值和失败判断统一按 `cg-staging-lighting.md`，本文件不维护第二套数字。
- 发现风险时，优先调整相机高度、角度、Look At 目标、Crane/Rail/Spline 路径；只有避障、前景控制或构图重锁确实需要时，才添加有职责的中间 Key。
- MCP 不能保证 100% 自动避障；结果报告写“基础避障检查/风险帧抽查”，不要承诺完全无碰撞。

## 运动质量验证

运动设计、Key 密度、曲线、Yaw 连续化和卡顿返工统一按 `professional-camera-keyframing.md`；Crane/Rail、焦距和焦点实现统一按 `ue5-camera-implementation.md`。本执行桥接只检查：

- Camera Path 控制方式与规格一致，没有 CineCamera 世界 Transform 和 Rig 参数双重控制。
- 每镜的位置、朝向和焦点各只有一个主控；控制方式和切换帧按 `ue5-camera-implementation.md` 读回，没有 Transform/Rig、Look At/Rotation 或 Manual/Tracking Focus 竞争。
- Key 数量由空间路径、节奏和控制方式决定；任何修正 Key 都能说明避障、构图、焦点或节奏职责。
- 读回主运动与高频层的轨道归属；周期性密集 Key 若分布在主 Transform 或无法单独关闭/删除，判为可编辑性失败，而不是高完成度。
- Dolly Zoom、推拉和近景主体距离变化时，焦距真实属性路径、Focus Method、语义焦点目标、`ManualFocusDistance`、光圈策略和镜头允许范围已正确绑定并读回。
- 当镜头依赖近景、长焦、浅景深、极端机位或关键肢体阅读时，先用代表姿势检查候选构图，再批量写完整路径；设计师明确要求的裁切、遮挡或抽象构图不因此判失败。
- 设计师要求落稳时，结尾应在 `EndFrame - 1` 之前进入最终构图；`EndFrame` 不属于 `[StartFrame, EndFrame)` 的有效画面。没有播放、录屏或渲染证据时只报告结构，不宣称流畅。

## Sequencer 范围与保存安全

- 所有 Playback Range、CameraCut 和 Section 在规格与报告中统一写成 `[StartFrame, EndFrame)`；持续帧数必须等于 `EndFrame - StartFrame`。执行 API 若使用其他边界语义，先转换再写入，读回后重新正规化报告。
- 修改 Playback Range 后，同步检查 CameraCut、Camera Transform、Actor Transform、FloatTrack 等 section range。
- 不要只相信 LevelSequence 时长。20 秒序列必须根据 Display Rate 计算目标持续帧数，并验证 Playback Range、CameraCut Range、关键轨道 Section Range 的 Start/End/Duration；例如 30fps 的 20 秒应为 `[0,600)`，最后有效显示帧为 599。
- 多 CameraCut 必须验证首段 Start、末段 End 和每段 Duration。普通硬切要求 `Previous.End = Next.Start`；设计师明确启用 `Can Blend` 时按计划重叠区间验收，并报告任何意外空洞或非计划重叠。
- 不要把 `EndFrame` 当作最后有效画面抽查；整数显示帧的结束画面应检查 `EndFrame - 1`，切镜边界同时检查 `EndFrame - 1` 与下一段 `StartFrame`。
- 重新 K 同一帧时，不要默认 AddKey 会替换旧 key；同一帧已有 key 应更新原 key 或先清理旧 key/旧 section。
- 重新制作同一动画时，优先清理旧轨道/旧 section，或使用 set-or-replace key 逻辑，避免重复 key 堆叠导致插值异常。
- 高层创建接口执行后，按 `ue5-sequencer-basics.md` 的“创建与幂等”先读回自动生成对象，再继续添加。
- 调用在写入后的验证、打印或序列化阶段报错，不代表此前修改已回滚。任何写操作异常后先读回目标对象；确认缺失后才重试，禁止直接整段重跑。
- MCP 超时、用户中断或客户端取消也不代表 UE 端写入已回滚；恢复后先读回资产、Binding、Track、Section 和 Key，只续写缺失部分。
- 复制或 Duplicate LevelSequence 后，区分 Spawnable 与关卡对象引用。轨道、Guid 和显示名被复制不代表对象仍有效；目标 Sequence 必须重新验证实际 Actor、CameraComponent 子 Binding 和父子关系，失效引用应在目标 Sequence 内重建，不沿用红色 Binding。
- 创建或修改 LevelSequence 后，必须保存 LevelSequence 资产包；保存当前关卡不等于保存序列资产。
- PIE 预览或设计师外部渲染前，先保存序列资产，再读回验证，避免渲染使用磁盘旧版本。

### CameraCut 完整性验证

- 默认只保留一个 CameraCut Track；存在多个 CameraCut Track 时必须说明用途，否则判为结构风险。
- 首段 CameraCut Start 必须等于 Playback Start，末段 End 必须等于 Playback End；任何设计师未要求的头尾空段都判为失败。
- 普通硬切的相邻 CameraCut 必须满足 `Previous.End = Next.Start`，并分别检查前一段 `EndFrame - 1` 和后一段 `StartFrame`。明确启用 `Can Blend` 时允许计划内重叠，改为验证重叠起止、两侧 Binding、混合期间 Transform 与参与的 CineCameraComponent 属性；合法 Blend 不按重复 Section 判失败。
- 每个 CameraCut Section 的 Binding ID 必须非空，并能解析到唯一、预期的 CineCameraActor；只看 Guid、显示名或 Section 存在不算验证成功。
- 同时验证 CineCameraActor 与 CameraComponent 子 Binding。Spawnable 尚未求值时可先验证模板、子 Binding 和类关系，再在低风险帧求值；复制 Sequence、红色 Binding 或关卡对象引用必须读回实际对象，不能以 Guid 匹配代替。
- 连续调用高层 `create_camera` 不保证自动产生对应数量的 CameraCut Section；多相机任务按计划镜头数显式读回、创建、分段和绑定，每段 Cut 只指向一台预期相机。
- 检查失效 Binding、重复 Section、零时长 Section、区间外 Section、未覆盖范围和错误相机复用。
- 新增、修改、复制、重绑或存在已知风险的 CameraCut / Binding，在保存并重新加载后重复读取 Track/Section、区间和 Binding 解析；未变化且已有有效读回证据的 Cut 不重复完整验收。标准制作和复制 Sequence 仍完整验证全部 CameraCut；内存中正确但重载后失效，判为失败。
- 如果连续 `[StartFrame, StartFrame+2)` 证明确有首帧错误求值，可在参与轨道写入 `StartFrame-1` 同值预滚 Key，并把轨道 Section 提前一帧；Playback 和 CameraCut 仍从 `StartFrame` 开始。没有首帧证据时不默认添加预滚。

## 执行与验证优化

目标是先快筛，再精查，避免定序器执行慢和高分辨率截图返工。

1. 结构验证：检查 Display Rate、Tick Resolution、Playback `[StartFrame, EndFrame)` 与 Duration、CameraCut 数量及连续性、摄像机绑定、关键轨道 Section Start/End/Duration、资产保存和读回。
2. 快速画面验证：优先抽查切镜两侧、速度或旋转峰值、最近障碍、焦点变化、动作关键姿势和结尾稳定区；没有识别出特定风险时再用起始/中段/结束前帧作低成本默认，不要一开始批量高分辨率截图。
3. 运动级验证：快速单镜头只做结构读回并交给设计师手动播放；快速动态默认在结构读回和低成本关键画面快筛后停止。只有已识别的具体风险、设计师要求或标准制作证据需要时，才按 `cg-preview-and-export-rules.md` 与 `mrq-temporal-diagnostics.md` 升级连续 PNG；烟雾测试通过且没有具体风险时，不自动扩大为多窗口或整段渲染。做不到时报告“不能确认运动流畅性”，不要卡死流程。
4. 修正问题：标准制作或成片检查发现黑屏、白屏、纯天空、纯地面、纯墙面、UI 外露、严重遮挡、语义为空、构图偏移、异常旋转、卡顿或顿挫时，先判断是相机本身、CameraCut 绑定、无目的中间 Key、Look At / Rotation 曲线，还是验证脚本/相机评估方式的问题。快速单镜头或快速动态的技术测试只有在异常妨碍当前技术目标验证时才返工；否则记录画面局限并停止，不自动追求正式镜头审美。
5. 正式检查：快筛和设计师运动验片通过后，再做成片检查或等待设计师提供正式渲染结果。

结构验证额外检查：

- CameraCut 时长和曲线形态只按设计师节奏、动作与技术目标判断；等长镜头或相近曲线本身不构成警告。
- 设置 Sequencer 插值、切线或 tangent mode 时，按通道类型或当前会话探测一次能力并复用结果，不对每个 Key 重复探测。调用可用方法，例如 `set_interpolation_mode`、`set_tangent_mode`、`set_arrive_tangent`、`set_leave_tangent`、`set_arrive_tangent_weight`、`set_leave_tangent_weight`；批量写入后抽样或按风险读回，只有异常、类型变化或会话重连时才重新探测。若目标通道不支持或读回不一致，使用不等距 Key 时间布局兜底，并在报告中说明未精确设置 Curve Editor。

验证截图注意：

- 如果脚本复现 Sequencer Transform，UE Python `Rotator` 构造顺序是 `roll, pitch, yaw`；Sequencer `Rotation.X/Y/Z` 对应 `roll/pitch/yaw`，不要按 `pitch/yaw/roll` 写。
- 如果整段截图倾斜、异常翻转或画面方向明显不符合预期，先怀疑验证脚本的旋转映射、相机评估方式或 CameraCut 取值，不要立刻判定设计师镜头失败。
- 高分辨率截图通常是异步落盘；不要一次请求大量截图后马上声明完成，必须确认文件存在并实际查看。
- 快速单镜头默认只做结构读回；快速动态、标准制作和成片预检按验收矩阵做对应深度的画面快筛。运动与成片结论必须有连续播放、录屏、渲染或等价时序证据；仅有数据和抽帧时不能升级结论。

## 画面级验证

除快速单镜头或用户明确要求只做结构验证外，按 `cg-preview-and-export-rules.md` 完成画面快筛；构图、遮挡、穿模、曝光和景深细则见 `cg-staging-lighting.md`。本文件不维护第二套验片清单。

执行层必须保证检查画面来自正式 CameraCut/CineCameraActor，并把关键帧截图、连续影像分析、设计师播放反馈和正式渲染证据分开报告。没有连续时序证据时，不能确认运动流畅性或成片质量。

## UE 播放/模拟状态安全

执行 Sequencer 写入、创建 CineCameraActor、修改 LevelSequence 或截图验证前，必须确认 UE 编辑器不在 PIE/Simulate 状态。

- 不要用 `unreal.EditorLevelLibrary.editor_play_simulate()` 来检查播放/模拟状态；它可能启动 Simulate/PIE，不是只读查询。
- 不要把带 `play`、`simulate`、`start`、`launch` 语义的 UE Python 函数当成状态检查。
- 如果不确定某个 API 是否只读，不要在执行前调用；优先使用只读状态查询，或让用户确认当前是否正在播放/模拟。
- 执行前先做 MCP 轻量响应检查；如果 UE/MCP 连续超时，不要继续写资产或反复重发脚本。
- 如果误触发 Simulate/PIE，立刻停止继续写入，让用户手动停止播放/模拟；停止后重新做 MCP 轻量连接检查，再从上一个安全阶段继续。

禁止把下面这句用于“检查是否正在播放”：

```python
unreal.EditorLevelLibrary.editor_play_simulate()
```

## UE5.8 Sequencer 预览安全

LevelSequence 创建后不要默认假设“能打开就稳定”。如果序列包含 Spawnable CineCameraActor、CameraCut、多段镜头切换，拖动 Sequencer 时间轴会触发 MovieScene 重新评估绑定和相机轨道，UE5.8 中可能崩溃。

默认流程：

- 不默认自动打开 Sequencer UI。
- 不要求用户创建后立刻拖动时间轴检查。
- 先通过脚本读回验证：Display Rate、Tick Resolution、Playback Start/End/Duration、CameraCut 数量、每段 `[StartFrame, EndFrame)` 与持续帧数、摄像机绑定是否有效。
- 验证通过后，再建议人工预览。

人工预览建议：

- 先从已经脚本读回的低风险关键帧开始播放或跳帧检查，例如开头、中段、切镜前后、结尾；不要宣称某一帧绝对安全。
- 避免连续快速拖动时间轴，尤其是多 CameraCut 和 Spawnable 摄像机序列。
- 如果要检查多段镜头，优先逐段跳到关键帧附近，而不是快速来回拖拽。

## UE Python 兼容注意

### 原生 API 路径
选择原生 API 或公开封装兼容失败时读 `ue-python-sequencer-fallback.md`。直接选择原生路径时按 live API 做 capability check；从失败路径切换时先读回半成品，只续写缺失部分，最后保存并按变化风险重载验证。

- 封装兼容错误不改变设计师镜头方案，也不是重建半成品的理由。
- 回退前必须确认现有 Sequence、Binding、Track 和 Section 状态；有效对象继续复用。
- Spawnable 尚未被 Sequencer 安全评估时，`get_bound_objects()` 可能为空；先验证 Binding、组件子 Binding、Property Track 和类默认值，只有需要运行时组件结论时才评估对应低风险帧。空对象列表本身不等于 Binding 失效。
- 回退后仍执行本文件的时间、CameraCut、控制权、参与焦点和保存验证；只有存在结束状态要求时才验证 Completion Mode。
- 只有状态未知、原生 API 不可用、必须修改插件/重启 UE，或继续操作可能破坏正式资产时才停止报告。


### CameraRig 执行索引

- Crane/Rail 的选择、对象、属性和最小落地方式统一见 `ue5-camera-implementation.md`；Key 密度、Rotation/Yaw 和双重控制统一见 `professional-camera-keyframing.md`。
- MCP 执行只额外验证 Rig Actor、CineCamera 挂载、Rig 主控轨道、CameraCut 目标和新增辅助资产报告。UE5.8 的 Rail + 动态 Look At 崩溃处理见 `cg-failure-cases.md`。
